# Lathe Architecture

## Core Principle

Smart decisions live in agent prompts. Dumb plumbing lives in the Go engine. When you see logic that *decides* something moving into engine code, that's drift.

## Two Layers of Prompts

**Meta-prompts** (`templates/meta-*.md`) — embedded in the binary, used only during `lathe init`. They instruct an AI how to analyze a project and write behavioral docs. These are the most important files in the repo. The quality of `templates/meta-goal.md` determines the quality of every goal-setter ever generated by lathe.

**Behavioral docs** (`.lathe/goal.md`, `.lathe/builder.md`, `.lathe/verifier.md`) — generated by init, read by runtime agents every cycle. These are project-specific. The templates produce them; they are outputs, not maintained by hand.

## Execution Flow

```
lathe init:
  meta-snapshot.md → runs agent → .lathe/snapshot.sh
  meta-goal.md     → runs agent → .lathe/goal.md + .lathe/skills/*.md + .lathe/alignment-summary.md
  meta-builder.md  → runs agent → .lathe/builder.md  (reads goal.md for alignment)
  meta-verifier.md → runs agent → .lathe/verifier.md (reads builder.md for failure modes)

lathe start (each cycle):
  goal.md + snapshot + goal-history → goal-setter → goal committed
  builder.md + snapshot + goal → builder → code committed
  verifier.md + snapshot + goal + diff → verifier → VERDICT: PASS or NEEDS_WORK
  loop builder/verifier up to roundsPerCycle (currently 4)
```

## File Map

```
main.go      — CLI entrypoint, global path vars, log/die helpers
init.go      — cmdInit, generateAgentRole, spinner, ensureGitignore, installSkill
engine.go    — engineStart (forks background process), engineStop, engineStatus, engineLogs
cycle.go     — runCycle (goal + adaptive builder/verifier), runStep (shared plumbing), readVerdict
agent.go     — runGoalSetter, runBuilder, runVerifier (assemble prompt, call invokeAgent)
invoke.go    — invokeAgent (calls claude/amp CLI, rate limit detection), sleepUntilRateLimitLifts
prompt.go    — assembleCommon (skills + refs + theme + snapshot), assembleSessionContext
snapshot.go  — collectSnapshot (runs .lathe/snapshot.sh), collectCIStatus
state.go     — Session/CycleState read/write, archiveCycle, archiveGoal, teardown
ci.go        — discoverPR, waitForCI, autoMergeIfGreen, captureCIFailureLogs
safety.go    — safetyNet (validates agent didn't do something catastrophic)
shell.go     — run, runCapture, runCaptureAll, runPipe, runPipeQuiet, runSilent
embed.go     — go:embed for templates/
```

## Key Invariants

- `.lathe/session/` is gitignored entirely — never blocks branch switches, never committed
- The engine runs as a background process (`lathe/_run`), not the foreground `lathe start` process
- Every step follows identical plumbing: branch → snapshot → CI status → agent → archive → safety net → push → PR → CI wait → merge
- Teardown works at any point — `lathe stop` kills the process tree, closes the PR, discards working tree changes, returns to base branch
- No fallback templates — init succeeds or fails cleanly

## Session State

`session.json` — branch name, PR number, base branch, mode (branch vs. direct), started_at
`cycle.json` — current cycle number, phase, status, updated_at
`snapshot.txt` — latest snapshot output (fed into every agent prompt, capped at 6000 chars)
`changelog.md` — latest agent changelog (verifier writes VERDICT here; engine reads it)
`goal-history/cycle-NNN.md` — goal-setter's output per cycle (goal-setter sees last 4)
`history/cycle-NNN/` — archived snapshots and changelogs

## Template Placeholders

`{{VALUES_MANIFESTO}}` — spliced into meta-goal.md from `templates/values-manifesto.md`
`{{INTERACTIVE}}` — spliced into meta-*.md when `--interactive` flag is set

## Build

Single Go binary, no external Go dependencies (only stdlib). Templates embedded via `go:embed`. Cross-compiled for darwin/linux arm64/amd64 and windows/amd64 by CI on push to main.

`go build ./...` — builds the binary
`go test ./...` — runs all tests (some tests use `t.TempDir()` and override global path vars)
`go vet ./...` — static analysis

## CI

One workflow: `.github/workflows/release.yml` — triggered by push to main only (not `pull_request_target` or `issue_comment`). Runs tests, builds 5 cross-compiled binaries, creates a GitHub release tagged with short SHA. No PR-triggered CI.
