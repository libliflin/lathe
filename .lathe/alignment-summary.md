# Alignment Summary

## Who This Serves

- **Lathe users** — solo developers and small teams running `lathe start` on their own projects, trusting the loop to improve their code without constant supervision
- **Contributors and maintainers** — people working on lathe itself, modifying Go engine code or meta-prompt templates
- **Init users** — anyone running `lathe init` for the first time on a new project; the alignment summary is their first signal of whether lathe understood their project
- **Future stakeholder sim** — the design in docs/next-session-prompt.md describes an aspirational model where a sim agent tries the project as a real user; this direction should be respected in architectural decisions

## Key Tensions

**Progress theater vs. genuine improvement** — The goal-setter is tempted to pick safe, visible, easy changes rather than the hard change that actually matters to a real user. Signal: goal history dominated by cleanup and polish while README's "Where This Is Heading" sits untouched.

**Current model stability vs. stakeholder sim model** — The builder/verifier loop works but the team is workshopping a richer sim-based model. Signal to favor current model: CI is red or the loop breaks. Signal to favor sim model: current model is stable and open questions in docs/next-session-prompt.md are tractable.

**Meta-prompt quality vs. engine code quality** — Meta-prompts are the highest-leverage thing in the repo but have no automated tests. Engine code is testable but is not the binding constraint. Signal: if agent logs show poor goal-setting or generic output, the meta-prompts need work, not the engine.

**Snapshot quality vs. snapshot complexity** — A good snapshot is a health signal, not a raw dump. The 6000-char cap means a noisy snapshot.sh pushes out important context. Signal: truncation warning in the snapshot means snapshot.sh needs rewriting.

## What Could Be Wrong

**Missing stakeholder: downstream projects that have already run lathe init** — If users are running lathe on their own projects and those projects have `.lathe/` directories committed, they are affected by changes to the meta-prompt templates. A change to `templates/meta-goal.md` affects every future `lathe init` run but doesn't retroactively update existing `.lathe/goal.md` files. This stakeholder group is real but invisible from the repo.

**No PR-triggered CI** — The GitHub Actions workflow only fires on push to main. There is no CI check on PRs before they merge. In branch mode, lathe merges its own PRs. If the CI workflow is the only gate and it only runs after merge, a bad commit goes to main before anyone can stop it. The goal-setter should watch for this: if the project moves to PR-based development with other contributors, this gap matters.

**Security posture is partial** — The repo is public. The CI workflow is not triggered by `pull_request_target` or `issue_comment` (good — these are injection vectors). The default branch protection status was not verified during init. The engine's prompt injection defenses (only reading structured fields from `gh`) are documented in the README but not enforced at a code level — they rely on the agent following instructions.

**Meta-prompt tests don't exist** — The most important outputs of this repo (the behavioral docs generated by `lathe init`) have no automated quality check. The goal-setter should consider whether there's a tractable way to test meta-prompt quality (e.g., running init against a known project and checking for required sections or absence of state-encoding antipatterns).

**Assumption about this being a lathe-on-lathe run** — Running lathe on lathe itself is unusual. The user's past guidance (in project memory) is "lathe is for simple sister projects with real stakeholders, never lathe-on-lathe." If that guidance is still operative, this init may be experimental and the goal-setter should be conservative about large structural changes to the meta-prompts without explicit user direction.
